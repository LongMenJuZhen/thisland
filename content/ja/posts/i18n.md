# はじめに

最近のAIコーディングアシスタントは本当に便利だね。ってことで、AIに手伝ってもらってこの博客を多言語対応にしたのだ。

ことの始まりは：Astro公式の国際化（i18n）チュートリアルを見つけたのだ。それで思った、「ブログもi18n対応にしちゃおうか？」と。中文でしか書いてないけど、いつか外国人も見るかもしれないよね？

それでAIにバリバリ指示を出して、公式ドキュメントを参考にもうなんてことなく実装しちゃったのだ。 물론 중간에 많은坑を踏んだけど、最後にちゃんとできたよ。

# なぜ国際化するの？

个人博客を国際化するのは一見オーバースペックに思える。でもよく考えて 보면：

1. **国際化ブログ ≠ 英語を書く**：中文でしか書かなくても、外国人に「ここに中文ブログがあるよ」と入口を与えられる
2. **技術記事の価値**：技術記事を書いてると国外の読者の方が多いこともある（英語技術コンテンツはもう太多了）
3. **SEOに良い**：Googleがもっと簡単に不同言語のコンテンツをインデックスできる
4. **勉強になる**：この機会にAstroのi18nメカニズムを学べる

# Astro i18nの核心概念

まず、Astroの国際化の核心概念を理解しよう。

## ルートi18n vs コンテンツi18n

Astroのi18nは実は二つのレイヤーがあるのだ：

### ルートi18n（astro.config.mjs）

これはURLレイヤーでの多言語サポート。比如：
- `/` → デフォルト言語（中国語）
- `/en/` → 英語
- `/ja/` → 日本語

`astro.config.mjs`に設定する：

```javascript
i18n: {
    locales: ["zh-cn", "en", "ja"],
    defaultLocale: "zh-cn",
    prefixDefaultLocale: false,  // デフォルト言語には接頭辞をつけない
    redirectToDefaultLocale: true,  // /にアクセスしたら/zh-cn/にリダイレクト
}
```

主要パラメータの解説：
- `locales`：BCP 47標準形式の 지원言語リスト
- `defaultLocale`：デフォルト言語
- `prefixDefaultLocale: false`：デフォルト言語は接頭辞なし（`/posts/xxx/`）、其他は接頭辞あり（`/en/posts/xxx/`）
- `redirectToDefaultLocale: true`：ルートパスにアクセスしたらデフォルト言語 首页に自動リダイレクト

### コンテンツi18n（Content Collections）

これは記事コンテンツの国際化。Astro公式には二種類の方案があるよ：

**方案一：フロントマターのlangフィールド**
```yaml
---
title: 私の記事
lang: en
---
```

全部の言語バージョンを同じディレクトリに入れ、`lang`フィールドで区別する。

**方案二：サブディレクトリ構造**
```
src/content/posts/
├── zh/           # 中国語記事
│   ├── post1.md
│   └── post2.md
├── en/            # 英語記事
│   └── post1.md   # post1の英語版
└── ja/           # 日本語記事
    └── post1.md
```

## なぜ方案二を選んだの？

最初はAIが方案一（フロントマターにlangフィールド）を 推荐したのだ、より简单だから。でも最終的に方案二を選んだ理由は：

1. **スラッグが独立**：方案一では不同言語の記事がスラッグを共有してしまう、方案二では独立命名 가능
2. **コンテンツ差が大きい時に柔軟**：中英語コンテンツが大きく異なる場合（これはよくある）、独立ファイルの方が便利
3. **冲突を避ける**：某記事在中には英語版がない場合、方案一は尴尬になる

## Astro.currentLocale vs ルート接頭辞

Astroテンプレートでは`Astro.currentLocale`で現在のページの言語を取得できる：

```astro
---
const locale = Astro.currentLocale; // "zh-cn", "en", または "ja"を返す
---
```

でも注意：**`Astro.currentLocale`は現在のルートの言語を教えてくれるだけで、直接コンテンツのフィルタリングには使えない**。なぜ？Astroの`getStaticPaths`はデフォルトでlocaleパラメータをfilter関数に渡してくれないからだよ。

# 核心の変更詳解

## 1. ディレクトリ構造の再編

まず記事を言語別にサブディレクトリに分類：

```bash
src/content/posts/zh/   # 中国語
src/content/posts/en/   # 英語
src/content/posts/ja/   # 日本語
```

でも这样做有一个问题：Astro会把子目录识别为独立的content collection！

```bash
# Astroは这些「自动生成」コレクションを生成してしまう：
# posts-zh, posts-en, posts-ja
# 而不是我们想要的统一的postsコレクション
```

**解决方案**：手動で`src/content.config.ts`を作成（注意点：`.`なし）、明示的にコレクションを定義する：

```typescript
import { defineCollection, z } from "astro:content";

const postsCollection = defineCollection({
    schema: z.object({
        title: z.string(),
        date: z.date(),
        lastmod: z.date().optional(),
        draft: z.boolean().optional().default(false),
        comments: z.boolean().optional().default(true),
        description: z.string().optional().default(""),
        image: z.string().optional().default(""),
        tags: z.array(z.string()).optional().default([]),
        category: z.string().optional().nullable().default(""),
        lang: z.string().optional().default(""),
    }),
});

export const collections = {
    posts: postsCollection,  // 自動生成されたposts-zh, posts-en, posts-jaを上書き
};
```

## 2. URL適応

これが一番面倒だった部分。中国語記事のURLは`/posts/xxx/`、英語は`/en/posts/xxx/`。

比如、`/posts/ASR/`を見ている時に言語切り替えボタンを押したら、`/en/posts/ASR/`に飛ぶべきなのだ。

### 問題その一：getStaticPathsがlocaleを渡さない

Astroの`getStaticPaths`はデフォルトでlocaleパラメータを渡してくれない：

```typescript
// ❌ 这样写localeはundefinedになる
export async function getStaticPaths({ locale }) {
    const posts = await getSortedPosts(locale);
}

// ✅ ハードコードする必要がある
export async function getStaticPaths() {
    const posts = await getSortedPosts("en");
}
```

つまり各言語ルート用に別々のページファイルを書く必要がある：
```
src/pages/
├── posts/[...slug].astro      # 中国語記事
├── en/posts/[...slug].astro   # 英語記事
└── ja/posts/[...slug].astro   # 日本語記事
```

### 問題その二：entry.slugが変わった

Astro 5のcontent collectionsでは、サブディレクトリ内のファイルのスラッグにはディレクトリ名が含まれる：

| ファイルパス | entry.slug |
|-------------|-----------|
| `posts/zh/ASR.md` | `zh/ASR` |
| `posts/en/ASR.md` | `en/ASR` |

これでは：
1. 中国語記事のURLが`/posts/zh/ASR/`になってしまう
2. 英語記事のURLが`/posts/en/ASR/`になってしまう

だから自前の関数を書いて純スラッグを抽出する必要がある：

```typescript
// src/utils/content-utils.ts

/**
 * entry.idから純スラッグを抽出（言語ディレクトリ接頭辞なし）
 * 例: "zh/ASR.md" → "ASR"
 */
export function getBaseSlug(id: string): string {
    const lastSlash = id.lastIndexOf("/");
    const filename = lastSlash >= 0 ? id.slice(lastSlash + 1) : id;
    return filename.replace(/\.mdx?$/, "");
}
```

それから`entry.slug`を使っている全部の場所で`getBaseSlug(entry.id)`に置き換える。

### 問題その三：言語スイッチャーのロジック

言語スイッチャーはこういう全部の場合を処理する必要がある：

| 現在パス | 英語に切り替え |
|---------|----------------|
| `/` | `/en/` |
| `/posts/ASR/` | `/en/posts/ASR/` |
| `/ja` | `/en` |
| `/ja/` | `/en/` |
| `/ja/posts/ASR/` | `/en/posts/ASR/` |

```typescript
// src/components/LanguageSwitch.svelte

function switchLocale(targetLocale: string) {
    const pathname = window.location.pathname;
    let cleanPath = pathname;

    // 既存の言語接頭辞を移除
    for (const loc of locales) {
        if (loc.code !== "zh-cn") {
            // /jaと/ja/の両方のケースを処理
            if (pathname === `/${loc.code}` || pathname.startsWith(`/${loc.code}/`)) {
                cleanPath = pathname.slice(`/${loc.code}`.length) || "/";
                break;
            }
        }
    }

    // 新URLを構築
    if (targetLocale === "zh-cn") {
        newUrl = cleanPath;
    } else {
        newUrl = `/${targetLocale}${cleanPath}`;
    }

    window.location.href = newUrl;
}
```

## 3. コンテンツフィルタリングロジック

言語別に記事をフィルタリングするのは核心機能：

```typescript
// src/utils/content-utils.ts

function getLangPrefixFromId(id: string): string {
    // IDから言語接頭辞を抽出
    // "zh/ASR.md" → "zh"
    const slashIndex = id.indexOf("/");
    return slashIndex >= 0 ? id.slice(0, slashIndex) : "zh";
}

function matchesLocale(id: string, locale?: string): boolean {
    const prefix = getLangPrefixFromId(id);
    // 統一化：zh-cn → zh, en → en, ja → ja
    const target = locale === "zh-cn" ? "zh" : locale || "zh";
    return prefix === target;
}

export async function getSortedPosts(locale?: string) {
    const allBlogPosts = await getCollection("posts", ({ id, data }) => {
        const draftFilter = import.meta.env.PROD ? data.draft !== true : true;
        return draftFilter && matchesLocale(id, locale);
    });
    // ... 排序等其他ロジック
}
```

## 4. 翻訳システムの改造

元の翻訳はグローバルだった：

```typescript
// ❌ 以前 - siteConfig.langだけを読み取る
export function i18n(key: I18nKey): string {
    const lang = siteConfig.lang;
    return getTranslation(lang)[key];
}
```

今は言語パラメータの受け渡しをサポート：

```typescript
// ✅ 今 - 動的言語をサポート
export function i18n(key: I18nKey, lang?: string): string {
    const targetLang = lang?.toLowerCase().replace("-", "_") || siteConfig.lang || "en";
    return getTranslation(targetLang)[key];
}
```

それから各コンポーネントで現在の言語を受け渡し：

```astro
---
const currentLocale = Astro.currentLocale || "zh-cn";
const homeLabel = i18n(I18nKey.home, currentLocale); // "首页"、"Home"または"ホーム"を返す
---
```

## 5. ページレベルのlocale処理

各言語のページではlocaleをハードコードする必要がある：

### 中国語首页 `src/pages/[...page].astro`
```typescript
export const getStaticPaths = (async ({ paginate }) => {
    const allBlogPosts = await getSortedPosts("zh-cn");  // ハードコード
    return paginate(allBlogPosts, { pageSize: PAGE_SIZE });
}) satisfies GetStaticPaths;

const currentLocale = Astro.currentLocale;  // 表示用に動的取得
```

### 英語首页 `src/pages/en/[...page].astro`
```typescript
export const getStaticPaths = (async ({ paginate }) => {
    const allBlogPosts = await getSortedPosts("en");  // ハードコード
    return paginate(allBlogPosts, { pageSize: PAGE_SIZE });
}) satisfies GetStaticPaths;

const currentLocale = "en";  // ハードコード
```

ここには小手先がある：`Astro.currentLocale`は静的ビルド中に確かに"en"を正しく返せるけど、明確さと曖昧さを避けるために、各言語ディレクトリ下のページではlocaleをハードコードすることにしたのだ。

## 6. 画像パス問題

記事の画像参照も修正が必要。例えば中文記事が`posts/zh/init.md`にあり、`../image/init/xxx.png`を参照：

```markdown
元：`posts/zh/init.md`で`![画像](image/init/xxx.png)`と書く
理由：`posts/`ディレクトリへの相対パス
直し：`posts/zh/init.md`で`![画像](../image/init/xxx.png)`と書く
理由：今は記事が`posts/zh/`にあるので、一つ上のレベルに行く必要がある
```

## 7. ナビゲーションリンク

ナビゲーションバーのリンクには言語接頭辞が必要：

```astro
---
const currentLocale = Astro.currentLocale || "zh-cn";
---

<!-- 以前 -->
<a href="/about/">关于</a>

<!-- 今 -->
<a href={`/${currentLocale === "zh-cn" ? "" : `/${currentLocale}`}/about/`}>关于</a>
```

或者、便利関数にラップ：

```typescript
// src/utils/url-utils.ts

export function url(path: string, locale?: string): string {
    const base = joinUrl("", import.meta.env.BASE_URL, path);
    if (!locale || locale === "zh-cn" || locale === "zh_CN") {
        return base;  // デフォルト言語には接頭辞なし
    }
    return joinUrl("", import.meta.env.BASE_URL, `/${locale.replace("_", "-")}`, path);
}
```

## 8. RSSとSitemap

### RSS

各言語には独立したRSSが必要：

```
src/pages/rss.xml.ts         → /rss.xml (中国語)
src/pages/en/rss.xml.ts      → /en/rss.xml (英語)
src/pages/ja/rss.xml.ts      → /ja/rss.xml (日本語)
```

### Sitemap

`@astrojs/sitemap`インテグレーションは自動的に多言語sitemapを処理してくれる、`i18n`オプションを設定するだけ：

```javascript
// astro.config.mjs
sitemap({
    i18n: {
        defaultLocale: "zh-cn",
        locales: {
            "zh-cn": "zh-CN",
            "en": "en",
            "ja": "ja",
        },
    },
}),
```

## 9. 検索適応

Pagefindは`data-pagefind-filter`属性で多言語インデックスをサポート：

```astro
<div data-pagefind-filter={`lang:${currentLocale}`}>
    <div data-pagefind-body>
        <!-- 記事コンテンツ -->
    </div>
</div>
```

検索時に言語でフィルタリング：

```typescript
const response = await window.pagefind.search(keyword, {
    filters: { lang: currentLocale }
});
```

## 10. 他に変更が必要だった場所

1. **サイドバーコンポーネント**：カテゴリ、タグリンクに言語接頭辞
2. **ページネーションコンポーネント**：改ページリンクに言語接頭辞
3. **PostCardコンポーネント**：記事カードリンクに言語接頭辞
4. **ArchivePanelコンポーネント**：アーカイブページを言語別にフィルタ
5. **Footer**：RSS、Sitemapリンクに言語接頭辞
6. **SEO**：`<html lang="...">`を正しく設定

# 最終結果

この博客は今三つの言語をサポートしている：

| 言語 | ルート | ナビゲーション文字 |
|------|--------|-------------------|
| 🇨🇳 中国語 | `/` | 首页、归档、关于 |
| 🇺🇸 英語 | `/en/` | Home、Archive、About |
| 🇯🇵 日本語 | `/ja/` | ホーム、アーカイブ、このサイト |

言語スイッチャーは右上の地球儀アイコンのボタン那里。

各言語の記事は独立して管理されている。中国語サイトには中国語記事だけ、英語サイトには英語記事だけが表示される。

# 結び

总的来说、AI辅助编程还是挺靠谱的，它可以帮你写代码，但关键的架构决策还是得自己来做。比如为什么选方案二不选方案一，为什么要在每个页面硬编码locale，这些都需要自己判断。

AI可以帮你：
- ✅ 写代码
- ✅ 解释概念
- ✅ 提供示例
- ✅ 帮你调试

AI不能帮你：
- ❌ 做架构决策
- ❌ 理解你的业务需求
- ❌ 判断什么是正确的方向

最後に、AIに、Claudeに感謝して、このブログはようやく「国際化」されたのだ（虽然现在英语記事只有那么几篇）。
